# Father List Bot — SaaS multibot build

Готовая SaaS-сборка Telegram-бота на `aiogram 2.25.1`.

Теперь это не просто один бот для публикаций, а **сервисный бот + клиентские боты на одном коде**:
- `@father_listbot` — сервисный бот владельца платформы
- клиентские боты — отдельные tenant-боты клиентов
- у каждого клиента свой бот, своё меню, свои каналы, свои шаблоны и своя база данных

## Что умеет система

### Сервисный бот
Сервисный бот используется как панель управления платформой:
- подключение клиентского бота по токену
- активация аренды клиентского бота
- продление аренды
- просмотр списка клиентских ботов
- запуск tenant runtime
- техническое управление SaaS-системой

### Клиентский бот
После подключения и активации клиент получает **своего отдельного бота**:
- админ-меню внутри своего бота
- свои шаблоны постов
- свои каналы для публикации
- свою загрузку медиа
- свою очередь публикации
- свои данные в отдельной SQLite-базе

### Изоляция данных
Для каждого клиента создаётся отдельная база:

```text
tenant_dbs/tenant_<owner_user_id>.sqlite3
```

Это значит:
- каналы одного клиента не видны другому
- шаблоны одного клиента не видны другому
- сервисный бот не смешивает клиентские посты с твоими
- клиент не получает доступ к админке сервисного бота

## Архитектура

### 1. Service bot
Главный бот платформы:
- обслуживает владельца сервиса
- принимает токены клиентских ботов
- активирует аренду
- управляет runtime клиентских ботов

### 2. Tenant bots
Каждый клиентский бот:
- запускается на том же проекте
- использует тот же код
- работает со своей tenant DB
- показывает админ-меню только владельцу этого бота

### 3. Разделение ролей
#### В сервисном боте
- `owner` — только владелец сервиса
- клиент-владелец tenant-бота здесь остаётся обычным `user`

#### В клиентском боте
- владелец этого бота получает роль `owner`
- работает уже только через своего бота

## Меню и роли

### Сервисный owner
В сервисном боте доступны:
- управление SaaS
- меню клиентских ботов
- активация на `30 / 60 / 90 / 180` дней
- просмотр и техническое управление

### Tenant owner
В своём клиентском боте владелец получает:
- создание и редактирование шаблонов
- предпросмотр поста
- добавление каналов
- загрузку медиа
- постановку в очередь
- управление публикацией
- рабочее админ-меню внутри своего бота

## Как это работает

### Подключение клиентского бота
1. Клиент заходит в `@father_listbot`
2. Подключает токен своего бота
3. Владелец сервиса активирует аренду
4. Система запускает tenant runtime
5. Клиент пишет `/start` уже своему боту
6. В своём боте клиент получает админ-меню

### Что важно
- клиент **не получает админку в `@father_listbot`**
- клиент получает админку **только в своём боте**
- сервисный бот и tenant-боты разделены

## Быстрый запуск

```bash
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
copy .env.example .env
python main.py
```

Или на Windows просто запусти `start_windows.bat` / свой launcher-файл.

## Обязательная настройка

1. Укажи сервисный токен в `.env`
2. Укажи свой Telegram ID в:

```env
SERVICE_OWNER_ID=ВАШ_ID
ADMIN_IDS=ВАШ_ID
```

3. Запусти проект
4. Открой `@father_listbot`
5. Подключи и активируй клиентский бот

## Минимальный `.env`

```env
BOT_TOKEN=YOUR_SERVICE_BOT_TOKEN
SERVICE_OWNER_ID=8292218374
ADMIN_IDS=8292218374
DEBUG_SAAS=0
```

## SaaS-структура данных

### Service DB
Сервисная база хранит:
- пользователей платформы
- данные клиентских ботов
- статусы аренды
- runtime-состояние
- системные настройки

### Tenant DB
Каждый клиент хранит отдельно:
- каналы
- шаблоны постов
- медиа
- очередь публикации
- настройки своего бота

## Активация клиентских ботов

В сервисном боте доступно меню:
- **🤖 Клиентские боты**
- **✅ Активировать**
- **⛔ Деактивировать**
- **🔄 Обновить список**

Срок активации:
- `30 дн.`
- `60 дн.`
- `90 дн.`
- `180 дн.`

## Диагностика

Для отладки можно включить:

```env
DEBUG_SAAS=1
```

После этого будет доступна команда:

```text
/debug_context
```

Она помогает проверить:
- `mode=service` или `mode=tenant`
- текущую роль пользователя
- какой бот обрабатывает сообщение
- какая база выбрана

## Важно

- Если бот клиента не активирован, tenant runtime не запустится.
- Если клиент сменил токен своего бота, токен нужно обновить в сервисном боте.
- Если бот не администратор в канале, публиковать он не сможет.
- Tenant DB хранится отдельно, поэтому данные клиента не смешиваются с данными платформы.

## Что уже решено в этой сборке

- клиентский бот отвечает **от своего токена**, а не от сервисного бота
- клиент получает меню владельца **в своём боте**
- в `@father_listbot` клиент остаётся обычным пользователем
- сервисный и tenant context разделены
- tenant runtime запускается отдельно для каждого клиента

## Поддерживаемые сценарии

- один сервисный бот + несколько клиентских ботов
- активация и продление аренды
- отдельная база для каждого клиента
- управление публикациями внутри tenant-бота
- debug-режим для SaaS-контекста

## Рекомендация по эксплуатации

После удачной настройки:
1. Сделай резервную копию `user_db.sqlite3`
2. Сделай резервную копию папки `tenant_dbs`
3. После этого уже добавляй новые функции: оплату, уведомления, обновление токена, автопродление, аналитику

## Short English summary

**Father List Bot SaaS build** is a multibot Telegram system where:
- one service bot manages the platform
- each client gets their own isolated tenant bot
- each tenant bot has its own SQLite database
- the client is admin only inside their own bot
- the service bot remains under the platform owner's control
